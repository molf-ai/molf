# 1. Mock & Harness Plan

## Current State Analysis

The test suite has a **massive duplication problem** with module mocking. The `packages/test-utils/src/mock-stream.ts` file already exports ready-to-use helpers (`mockAiModule`, `mockGoogleModule`, `mockAnthropicModule`, `mockProviderRegistryModule`) â€” but **zero test files use them**. Every test file that needs LLM mocking manually inlines the same `mock.module()` boilerplate.

Additionally, 8 test files define an identical local `makeStream()` function that is functionally identical to the already-exported `mockStreamText()` from test-utils.

### Why test files don't use the existing helpers

The root cause is Bun's `mock.module()` timing constraint: mocks must be set up **before** any `import` of the modules they replace. Because `mock.module()` callbacks are evaluated lazily (on first import), the mock setup itself cannot import from `@molf-ai/test-utils` â€” that would trigger side-effect imports. The current pattern uses `await import(...)` **after** `mock.module()` calls. Since `mock.module()` must appear at the top-level before imports, test files cannot do:

```ts
// CAN'T DO THIS â€” the import happens before mock.module() is registered
import { mockAiModule } from "@molf-ai/test-utils";
mock.module("ai", () => mockAiModule(streamTextImpl));
```

They also can't use `await import("@molf-ai/test-utils")` inside the `mock.module()` factory because the factory must be synchronous. The `let streamTextImpl` closure pattern further complicates things: the mock factory captures a mutable variable that tests reassign per-test, and this closure must be set up before any SUT import.

## Proposed Harness Design

> **Resolved per Devil's Advocate [BLOCKING] #1**: Use a shared harness module imported directly instead of preload + `globalThis`. See [Resolution Log](#6-resolution-log).

### Strategy: Shared harness module (no preload, no globalThis)

Create a **shared harness file** that test files import at the top level. The harness file calls `mock.module()` at module evaluation time and exports a `setStreamTextImpl` setter. Because `mock.module()` is evaluated when the harness is imported (before `await import()` of the SUT), the timing constraint is satisfied.

#### File to create: `packages/test-utils/src/ai-mock-harness.ts`

```ts
import { mock } from "bun:test";

let streamTextImpl: (...args: any[]) => any = () => {
  throw new Error("streamTextImpl not set â€” assign it in beforeEach");
};

mock.module("ai", () => ({
  streamText: (...args: any[]) => streamTextImpl(...args),
  tool: (def: any) => def,
  jsonSchema: (s: any) => s,
}));

mock.module("@ai-sdk/google", () => ({
  createGoogleGenerativeAI: () => () => "mock-model",
}));

mock.module("@ai-sdk/anthropic", () => ({
  createAnthropic: () => () => "mock-model",
}));

export function setStreamTextImpl(impl: (...args: any[]) => any): void {
  streamTextImpl = impl;
}
```

#### File to modify: `packages/test-utils/src/index.ts`

Add export:
```ts
export { setStreamTextImpl } from "./ai-mock-harness.js";
```

### Why this works

- `import { setStreamTextImpl } from "@molf-ai/test-utils/ai-mock-harness"` is a static import, evaluated first
- The import triggers the harness module's top-level `mock.module()` calls
- `await import("../src/agent.js")` runs after, so mocks are already registered
- The `streamTextImpl` closure lives in the harness module, not `globalThis`
- No `bunfig.toml` change needed, no preload, no hidden side-channels

### Why NOT preload + globalThis

The Devil's Advocate identified that `globalThis` is a hidden global side-channel with no type safety. The shared harness module achieves the same deduplication with:
- Better locality (import visible in file)
- Better debuggability (follow imports, not globalThis)
- No bunfig.toml changes
- Module-scoped closure instead of global state

- **Reporter**: JUnit XML output not needed unless CI system requires it. GitHub Actions gets annotations from Bun automatically.

**Recommended CI script change:**

# 6. Resolution Log

## [BLOCKING] #1: globalThis side-channel for preload (Section 1)

**Concern**: The original proposal used `(globalThis as any).__streamTextImpl` to bridge mutable state between a preload file and test files. This is a hidden global side-channel with no type safety.

**Resolution**: Replaced with a **shared harness module** (`packages/test-utils/src/ai-mock-harness.ts`) that test files import directly. The `streamTextImpl` closure lives in the module scope, not `globalThis`. No preload needed. No `bunfig.toml` changes. The import is visible in each test file, making the mock dependency explicit.

**Impact**: Eliminates 36 of 42 `mock.module()` calls (same as original proposal) with better debuggability and no hidden state.

## [BLOCKING] #2: Merge server-worker.test.ts into full-flow.test.ts (Section 4)

**Concern**: The overlap between these files is in test *subjects* (session CRUD), not test *code*. Merging would increase blast radius of failures, create a larger fixture setup, and produce churn for marginal benefit. 155 lines is not a large file.

**Resolution**: Keep both files. Testing the same API from different angles is defense in depth, not waste. If `tool.approve/deny` stubs need relocation, move them to `tool-approval.test.ts` as a standalone change.

## [BLOCKING] #3: Split router-edge-cases.test.ts into 6+ files (Section 4)

**Concern**: Distributing 12 test groups across 6+ different files is high-risk:
- Each destination file has its own setup, mock state, and lifecycle hooks
- Moving tests that need slow mocks to mock-free files requires adding mock infrastructure
- A single "edge cases" file has documentation value â€” it's where developers look for boundary behavior

**Resolution**: Rename to `server-api-edge-cases.test.ts` and add section comments. Split incrementally only when individual test groups grow large enough to warrant their own file. This preserves the shared mock setup and provides a clear structure without the migration risk.

# 7. Additional Gaps (Post-Review)

> Identified during secondary analysis. These gaps were not covered by the original 5-agent review.

## 7.1 Async cleanup not awaited in test helpers

**Files**: `packages/e2e/helpers/test-server.ts`, `packages/e2e/helpers/test-worker.ts`

`cleanup()` is synchronous, but `instance.close()` may be async internally. `tmp.cleanup()` runs immediately after, while the server may still hold file handles on session/upload directories.

```ts
// Current (test-server.ts)
cleanup() {
  instance.close();   // possibly async
  tmp.cleanup();      // runs immediately â€” race condition
}
```

**Fix**: Make `cleanup()` async and await `instance.close()` before deleting the temp directory. This is also a production improvement â€” `startServer().close()` should return a Promise for graceful shutdown.

## 7.2 Timer and subscription leaks in prompt helpers

**Files**: `packages/e2e/helpers/prompt-helpers.ts`, `packages/e2e/helpers/wait-for-event.ts`

If a test times out externally (Bun's per-test timeout) before `promptAndWait` resolves, the internal `setTimeout` and tRPC subscription remain active. The timer fires during the next test, potentially causing spurious failures.

Similarly, `waitForEvent()` creates a timer that is never cleared if the returned Promise is abandoned.

**Fix**: Use `AbortController` or a cleanup callback pattern so that dangling timers and subscriptions are cancelled when the test ends. Alternatively, ensure `afterEach` hooks unsubscribe all active subscriptions.

## 7.3 EventBus listener leak in concurrent-sessions.test.ts

**File**: `packages/e2e/tests/integration/concurrent-sessions.test.ts`, lines 71-89

The "rapid subscribe/unsubscribe" test calls `unsub()` only on even iterations, leaving 10 listeners subscribed to the EventBus. No cleanup in afterAll.

**Fix**: Track remaining subscriptions and unsubscribe all in the test's finally block (or afterEach). This is separate from the weak assertion fix (item 7 in Section 4).

## 7.4 Dead code: `waitForEvent()` helper

**File**: `packages/e2e/helpers/wait-for-event.ts`

Not imported by any test file. All e2e tests use `promptAndWait`, `collectEvents`, or `waitUntil` instead.

**Fix**: Delete the file and remove its export from `packages/e2e/helpers/index.ts`.

## 7.5 Inconsistent `test()` vs `it()` naming

`packages/client-telegram/tests/` uses `it()` (254 calls). All other packages use `test()`.

**Fix**: Standardize on `test()` across the codebase. Low priority but high volume â€” do as a single bulk rename.

## 7.6 Console mock without centralized cleanup

**Files**: `packages/client-telegram/tests/bot.test.ts`, `packages/client-telegram/tests/handler.test.ts`, `packages/protocol/tests/cli.test.ts`

6+ instances of manually saving/restoring `console.log` and `console.error` via try/finally. If the test framework doesn't execute finally (e.g., process crash), console remains mocked.

**Fix**: Use `mock.module` or `spyOn(console, "log")` via `bun:test` instead â€” Bun auto-restores spies between tests.
